## Playbooks

Playbooks are **YAML files used to automate tasks**.

- Playbook is a collection of multiple modules
- We can execute all the modules written inside a playbook at a time
- Playbook is also written using yml
- YML starts with --- and ends with ... (Both of these are optional)
- Playbook contains tasks

basic syntax of a playbook:

```yaml playbook.yaml
- name: Name of the playbook
  hosts: group_name # group name defined in the inventory file or all for all the hosts
  become: yes/no (if you want to run the tasks with sudo privileges)
  
  tasks:
    - name: Name of the task
      module_name:
        key1: value1
        key2: value2
```

1. Check the syntax of the playbook before running it:

```bash
ansible-playbook playbook.yaml --syntax-check
```

2. Run the playbook:

```bash
ansible-playbook playbook.yaml
```

Note: `Gathering facts` is enabled by default in playbooks. It collects information about the remote hosts before executing the tasks. This can be useful for conditional tasks or when you need to use the collected information in your playbook. However, if you want to disable fact gathering to speed up the execution of your playbook, you can set `gather_facts: no` at the play level. It will run before the tasks and will not collect any information about the remote hosts. This can be useful when you don't need the collected information for your tasks and want to reduce the execution time of your playbook.

3. Playbook to install apache on all the worker nodes:

```yaml install_apache.yml
- name: Install Apache on worker nodes
  hosts: all ## group name defined in the inventory file or all for all the hosts
  become: yes # switch to root user to install packages
  tasks:
    - name: Install Apache
      yum:
        name: httpd ## package name
        state: present ## state can be present, absent, latest
    - name: print a message
      debug: ## module to print a message
        msg: "Apache has been installed successfully!"
    - name: Start Apache service
      service:
        name: httpd
        state: started
        enabled: yes
```

4. Installtion of git ,maven, docker using a playbook:

```yaml install_packages.yml
- name: Installtion of git ,maven, docker
  hosts: all
  become: yes
  tasks:
    - name: Install git
      yum:
        name: git
        state: present
    - name: Install maven
      yum:
        name: maven
        state: present
    - name: Install docker
      yum:
        name: docker
        state: present
    - name: start docker service
      service:
        name: docker
        state: started
        enabled: yes # enable docker service to start on boot
```

to uninstall the packages, you can change the state to absent in the playbook:

```yaml
state: absent
```

When we have multiple tasks in a playbook editing a playbook is not recommended , so we can run a linux command to uninstall the packages using the command module:

```bash
sed -i 's/present/absent/g' install_packages.yml
```
 
5. Installation of nginx using a playbook:

```yaml install_nginx.yml
- name: Install nginx on worker nodes
  hosts: all
  become: yes # switch to root user to install packages
  tasks:
    - name: enable nginx repository ## only for amazon linux 2 because nginx is not available in the default repository of amazon linux 2 or older versions of linux
      command: sudo amazon-linux-extras enable nginx1  

    - name: Install nginx
      yum:
        name: nginx
        state: present
    - name: Start nginx service
      service:
        name: nginx
        state: started
        enabled: yes # enable nginx service to start on boot [automatically start nginx service when the server is rebooted]
```

6. Deploy a nginx web page using a playbook:

```yaml deploy_nginx_page.yml
- name: installation of nginx and starting nginx service
  hosts: all
  become: yes
  tasks:

    - name: install nginx
      yum:
        name: nginx
        state: present

    - name: start nginx
      service:
        name: nginx
        state: started
        enabled: yes

    - name: deployment of web page
      copy: # module to copy files from local to remote
        src: index.html # source file on the ansible master node
        dest: /usr/share/nginx/html/index.html # destination file on the worker nodes
        mode: '0644' # file permissions

    - name: restarting nginx service
      service:
        name: nginx
        state: restarted
```

## Project 1: Executing a Shell Script (Sample Nginx App) using Ansible Playbook on "Dev-Servers"
=============================================================
1) Configure "dev-servers" in Ansible Host Inventory File

2) In the Ansible Master Node, be in /root path

3) Create a Sample Shell Script

vi deploy_app.sh ----->

```bash
#!/bin/bash

echo "Starting sample application deployment..."

# Update packages
sudo yum update -y

# Install nginx as a sample application
sudo yum install -y nginx

# Start nginx service
sudo systemctl enable nginx
sudo systemctl start nginx

echo "Deployment completed successfully!"
```
3.1) Make the script executable:
chmod +x /root/deploy_app.sh

4) Ansible Playbook to Execute the Script

vi run-deploy.yml ---->
```yaml
- name: Run deployment script on dev servers
  hosts: dev-servers
  become: yes

  tasks:
    - name: Copy deployment script to remote server
      copy:
        src: /root/deploy_app.sh
        dest: /tmp/deploy_app.sh
        mode: '0755'

    - name: Execute deployment script
      command: /tmp/deploy_app.sh
```

5) Run the playbook

```bash
ansible-playbook run-deploy.yml
```

This will:
✔ Copy your shell script from the master (controller)
✔ Place it on /tmp/deploy_app.sh on each dev server
✔ Make it executable
✔ Run it normally

🔥 Result
Every time you run the playbook, your script automatically executes on all dev-servers exactly as you want.

6) To Remove Ansible using AdHoc Command

```bash
ansible dev-servers -b -m yum -a "name=nginx state=absent autoremove=yes"
OR
ansible dev-servers -b -m yum -a "name=nginx state=absent"
```

✅ Explanation of -b, -m, and -a
-b → Become (sudo)

Same as writing: --become

If you remove -b, Ansible will try to remove nginx without sudo, which will fail.

So YES — it is required for package operations.

-m → Module

-a → Arguments

-a means:

➡️ Pass arguments to the module
Here, 
"name=nginx state=absent" are the arguments expected by the yum module.

If you remove -a, Ansible will not know what to uninstall.

7) To Remove Ansible using Playbook (Ask Students to Write)
```yaml
- name: Remove nginx from dev servers
  hosts: dev-servers
  become: yes

  tasks:
    - name: Stop nginx service if running
      service:
        name: nginx
        state: stopped
      ignore_errors: yes

    - name: Disable nginx service
      service:
        name: nginx
        enabled: no	#Disable the service, so it will not start again
      ignore_errors: yes

    - name: Remove nginx package
      yum:
        name: nginx
        state: absent	#uninstall the package
        autoremove: yes	#remove unused dependencies
```
Execute the Playbook;
ansible-playbook remove-nginx.yml

### SETUP Module

Setup module will show the system info about the worker nodes

```bash
ansible all -m setup
```

Only CPU information of the worker nodes:

```bash
ansible all -m setup | grep -i cpu
or 
ansible all -m setup -a "filter=ansible_processor*"
```

only memory information of the worker nodes:

```bash
ansible all -m setup -a "filter=ansible_mem*"
```
worker node belong to which family:

```bash
ansible all -m setup | grep -i family
or
ansible all -m setup -a "filter=ansible_os_family"
```

Printing a message using debug module:

```yaml
- name: Print a message
  hosts: all
  tasks:
    - name: Print a message using debug module
      debug:
        msg: "- name: printing messages
  hosts: all
  tasks:
    - name: printing the message
      debug:
      msg: "number of cpus: {{ ansible_processor_vcpus }}, totalmemsize: {{ ansible_memtotal_mb }}, the flavour of linux: {{ ansible_os_family }}"
```
ansible_memtotal_mb how can we get this variable?

ansible_memtotal_mb is a variable that is automatically collected by the setup module when you run the command `ansible all -m setup`. It contains the total memory size of the worker nodes in megabytes. You can use this variable in your playbooks or ad-hoc commands to access the total memory information of the worker nodes.
