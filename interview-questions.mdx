# Ansible Mock Interview (Q&A with Detailed Explanations)

# 🔹 SECTION 1: CORE CONCEPTS


## 1. What is Ansible?

Ansible is an open-source automation tool used for configuration management, application deployment, and infrastructure provisioning. It is agentless and uses SSH to communicate with nodes.

**Explanation:**
Key points interviewer expects:

* Agentless (no installation on nodes)
* YAML-based (easy to read)
* Push-based (control node pushes configs)


## 2. How does Ansible work internally?

Ansible control node connects to managed nodes via SSH, executes modules, and returns results.

**Explanation:**
Flow:

1. Playbook → API
2. API → SSH
3. Module runs → JSON output


## 3. What is an inventory file?

It defines the list of hosts and groups that Ansible manages.

**Explanation:**
Types:

* Static (manual IPs)
* Dynamic (cloud-based)


## 4. Ad-hoc vs Playbooks?

Ad-hoc commands are one-time quick tasks, while playbooks are reusable YAML files for automation.

**Explanation:**
Ad-hoc → quick fix
Playbook → production automation


## 5. Difference between command and shell module?

Command does not support shell features, while shell supports pipes, redirects, etc.

**Explanation:**
Use command → safer
Use shell → only when needed


## 6. What are modules?

Modules are reusable units of work like installing packages or managing services.

**Explanation:**
Examples:

* apt
* yum
* copy
* service


## 7. What is a playbook?

A YAML file that defines automation tasks to be executed on hosts.


## 8. What is variable precedence?

Order in which variables override each other.

**Explanation:**
Highest → Lowest:

* extra-vars
* play vars
* inventory vars
* role defaults


## 9. What are handlers?

Handlers are tasks that run only when notified by other tasks.

**Explanation:**
Used for actions like restarting services only when config changes.


## 10. What is idempotency?

Running a playbook multiple times produces the same result without unnecessary changes.

**Explanation:**
This ensures safe and predictable automation.


## 11. How does Ansible ensure idempotency?

By using modules that check the current state before making changes.


## 12. What is Jinja2 templating?

A templating engine used to create dynamic configuration files using variables.

**Explanation:**
Example:
{{ variable_name }}


## 13. copy vs template?

Copy transfers static files, template processes dynamic files using Jinja2.


## 14. What are tags?

Tags allow selective execution of tasks in a playbook.


## 15. What is dynamic inventory?

Automatically fetches hosts from cloud providers instead of static files.


## 16. What is ansible.cfg?

Configuration file that controls Ansible behavior.


## 17. What is become?

Privilege escalation mechanism to run tasks as another user.


## 18. What are roles?

Reusable and structured way of organizing playbooks.


## 19. What is Ansible Vault?

Used to encrypt sensitive data like passwords.


## 20. What are async tasks?

Tasks that run in the background without blocking execution.


# 🔥 SECTION 2: REAL-WORLD / SCENARIO QUESTIONS


## 21. Your playbook failed midway. What will you do?

Use block, rescue, and always for error handling.

**Explanation:**

* block → main tasks
* rescue → runs on failure
* always → cleanup/logging


## 22. Works in dev but fails in prod?

Check environment differences, permissions, variables, and dependencies.


## 23. Different configs for dev/staging/prod?

Use inventory groups and group_vars with Jinja2 templates.


## 24. Update 100 servers gradually?

Use serial keyword.

**Example:**
serial: 10
If you have 100 servers:

First → updates 10 servers
Then → next 10
Continues batch by batch

## 25. Improve performance of playbook?

Increase forks, use async, and free strategy.


## 26. Debug failed playbook?

Use -vvv flag and debug module.


## 27. Ensure idempotency in real projects?

Avoid shell commands, use proper modules.


## 28. Manage secrets securely?

Use Ansible Vault.


## 29. Integrate Ansible with CI/CD?

Use Jenkins or pipelines to trigger playbooks.


## 30. Explain a real project using Ansible

Automated deployment using:

* Dynamic inventory (AWS)
* Roles for structure
* Jinja2 templates for configs
* Tags for selective execution
* Error handling for stability
* Idempotency for reliability


# ⚡ SECTION 3: TRICKY QUESTIONS


## 31. What happens if a handler fails?

Playbook stops unless handled with error handling.


## 32. Can Ansible run without SSH?

Yes, using local or other connection plugins.


## 33. include_tasks vs import_tasks?

- import_tasks = Static (decided before execution)
- include_tasks = Dynamic (decided during execution)


## 34. What is fact gathering?

Collects system info before tasks.


## 35. How to disable fact gathering?

Set:
gather_facts: no


## 36. What is fork in Ansible?

Number of parallel processes.


## 37. Difference between roles and playbooks?

Roles = reusable
Playbooks = execution logic


## 38. What is lookup plugin use case?

Fetching external data (files, env vars).


## 39. Why avoid shell module?

Not idempotent and less secure.


## 40. Biggest challenge you faced in Ansible?

Explain a real issue + how you debugged + fixed it.
